### Claude Cowork SharedRoot: 로컬 AI 에이전트 VM에서 guest-root·host mount를 함께 차단해야 하는 이유

핵심: Accomplish AI 연구를 인용한 보도에서 Claude Cowork의 local session이 Linux guest kernel `CVE-2026-46331`을 통해 guest-root를 얻으면 read-write로 mount된 Mac 전체 파일시스템에 도달할 수 있는 경로 공개. 대응: 최신 Cowork의 cloud execution 기본값과 별개로 local execution 사용 조직은 전체 host mount·unprivileged user namespace·module autoload·`coworkd` 권한을 즉시 점검 필요.

#### 원문과 관련 뉴스의 핵심 내용

Accomplish AI가 Claude Cowork local session의 VM sandbox escape를 SharedRoot로 명명해 공개했고, The Hacker News가 2026년 7월 23일 이를 보도함.

- 발행: The Hacker News RSS `pubDate` 2026-07-23 18:57:59 +05:30을 **2026-07-23 22:27 KST**로 변환
- 영향 주장: 연구진은 patch 이전 local Cowork macOS 사용 약 50만 명을 영향 가능 범위로 제시
- 경계: 보도·연구는 local session 경로를 다루며 모든 Claude 제품·모든 macOS endpoint의 침해 사실을 뜻하지 않음

#### 공격 체인

Cowork desktop app은 로그인 사용자 권한으로 동작하고 agent 작업은 Apple Virtualization framework의 Linux VM에서 처리됨. 연결 폴더 공유를 맡는 `coworkd`가 host filesystem을 VM으로 제공하는 구조가 후속 경계가 됨.

- 전제: guest의 unprivileged user namespace에서 `CAP_NET_ADMIN`을 얻는 namespace 구성
- 권한 상승: Linux `tc/act_pedit` 경로의 `CVE-2026-46331`을 악용해 guest-root를 얻는 연구 chain
- 전이: guest-root가 `/mnt/.virtiofs-root`의 read-write host `/` mount에 도달하는 구조

#### 영향 범위와 증거 경계

guest-root 이후 host 파일시스템이 read-write로 노출돼 있으면 로그인 사용자 권한으로 접근 가능한 SSH key·cloud credential·소스 파일이 위험 범위에 들어갈 수 있음. 이는 연구가 제시한 조건부 경로이며 조직별 노출은 local mode·kernel·mount 설정으로 확인 필요.

- 포함 가능 자산: user home의 source tree·SSH key·cloud credential·application config
- 미확정: 공개 보도만으로 특정 기업의 악용·침해 계정·공격 campaign 확인 불가

#### AI/Infra 전문가 해석

이 사례의 통제 실패 지점은 VM 내부 non-root user만이 아니라 guest-root 뒤의 host mount와 privileged daemon 경계임. agent sandbox는 kernel patch 주기보다 host resource visibility와 write capability를 먼저 축소해야 함.

- 원칙: guest compromise를 가정해도 host `/`가 보이지 않도록 shared path를 최소화
- 관측: VM process·mount table·namespace·module load·`coworkd` child process를 endpoint timeline으로 연결

#### 보안·운영 관점

Anthropic이 최신 Cowork에서 cloud execution을 기본값으로 둔다고 보도됐으나, local execution opt-in의 runtime policy와 endpoint kernel 상태는 별도 자산·변경 관리 대상임.

- 정책: local mode 허용 사용자·업무·비밀정보·network egress의 승인 기준 분리
- 패치: guest kernel·host app version 확인은 필요하지만 host-wide write mount가 남으면 구조적 위험이 지속되는 조건

#### 개발팀/인프라팀 체크리스트

local Cowork를 쓰는 Mac은 cloud default 전환 여부와 무관하게 실제 session mode·mount scope·kernel 설정을 확인해야 함.

- inventory: Cowork version·local session opt-in·연결 folder·developer credential 보유 endpoint 대조
- hardening: host `/` 대신 연결 folder만 mount하고 read-only 가능성 검토
- hardening: unprivileged user namespace·module autoload·과도한 seccomp allowlist 비활성화 또는 제한
- hardening: `coworkd`의 별도 mount namespace와 `ProtectSystem=strict` 적용 가능성 검토

#### 강조 메세지

VM sandbox는 guest user 격리만으로 평가할 수 없음. guest-root 이후 host filesystem과 daemon을 통해 어디까지 갈 수 있는지가 실제 blast radius를 결정함.

- 발표 포인트: **host `/` read-write mount**가 kernel privilege escalation을 Mac 사용자 파일 접근으로 전이하는 구조
- 팀 액션: local agent 실행은 secret 없는 격리 계정·최소 mount·endpoint telemetry를 갖춘 canary에서만 검증

### 에이전틱 AI 이니셔티브: 국내 agent 도입에서 권한·신원·감사·실행 비용을 운영 계약으로 만드는 방법

핵심: 과기정통부가 에이전틱 AI 안전·신뢰, 개방형 실행 기반, 수요 중심 도입을 3대 추진 전략으로 제시하고 연내 안전·신뢰 가이드라인·에이전트 신원·책임성 검토를 예고. 운영: AI OS·마켓플레이스·멀티에이전트 실행 지원은 실제 서비스 배포 기준이 아니므로 권한·데이터·로그·token 비용·책임 경계를 조달·PoC 조건으로 구체화 필요.

#### 한 줄 결론

국내 에이전틱 AI 정책은 안전·신뢰와 개방형 실행 생태계를 함께 다루기 시작했지만, 서비스 도입의 실제 안전성은 agent identity·tool authorization·data egress·audit·rollback을 실행 가능한 통제로 만드는 데 달려 있음.

- 결정: 정책 지원·실증 기회와 production control evidence를 같은 기준으로 취급하지 않는 원칙
- 팀 액션: PoC 승인 전 side effect·권한·데이터·비용·사고 대응 owner를 명시

#### 원문과 관련 뉴스의 핵심 내용

과기정통부는 7월 23일 에이전틱 AI 이니셔티브를 발표하고 안전·신뢰, 개방형 실행 기반, 수요 중심 도입을 3대 전략으로 제시함. 인공지능신문 기사 입력 시각은 13:04 KST임.

- 연내 계획: 개인정보·민감 데이터·AI 권한·목표 이탈을 포함한 안전·신뢰 가이드라인 마련
- 검토: 에이전틱 AI 얼라이언스에서 agent identity와 행위 책임 소재 검토
- 경계: 법적 의무·기술 표준·예산·대상·시행일은 보도 범위를 넘어 확정하지 않음

#### AI/Infra 전문가 해석

agentic AI는 model endpoint가 아니라 model·tool·workflow·identity·audit가 연결된 control plane임. 실행 권한을 가진 agent는 응답 품질보다 side effect의 대상·승인·복구 가능성이 운영 위험을 결정함.

- 아키텍처: agent service account와 human identity·tool credential을 분리
- 관측: prompt·policy decision·tool call·result·human approval·egress를 하나의 trace ID로 연결
- release: model 변경보다 tool schema·permission·fallback의 regression test가 우선

#### 안전·거버넌스 관점

정책이 권한 오남용·보안 위협·민감 데이터 유출을 위험 요소로 명시한 만큼, 조직은 abstract safety claim 대신 권한 범위와 책임 소재를 사후 재현 가능한 control로 바꿔야 함.

- 권한: read·write·purchase·deploy·delete를 별도 capability로 분해하고 고위험 action은 independent confirmation 적용
- 책임: agent identity·tool identity·approver·change ticket을 immutable audit trail에 보존
- 데이터: source provenance·classification·retention·external API egress를 tool policy 전에 평가

#### 비용·운영 관점

정책이 token 효율 기반 실행 운영 기술을 언급한 것은 multi-agent와 반복 tool use의 운영비가 모델 가격표만으로 설명되지 않는다는 문제와 연결됨.

- FinOps: token·cache·queue·tool retry·human review·external API cost를 성공 작업당 비용으로 집계
- SRE: task completion·p95 latency·unsafe-action reject·fallback·manual handoff를 같은 SLO로 관리
- 용량: GPU 지원 발표와 서비스별 quota·region·SLA·network·observability 요구를 분리 검증

#### 도입 로드맵

실증은 넓은 업무 자동화가 아니라 permission이 좁고 rollback 가능한 workflow부터 시작해야 함. 단계별 evidence가 확보될 때만 agent autonomy와 tool scope를 확장하는 방식이 적절함.

- 1단계: data flow·tool inventory·권한·owner·incident path를 문서화하고 no-op 또는 read-only canary 구성
- 2단계: bounded write action에 human approval·policy log·rollback·kill switch 연결
- 3단계: representative task에서 품질·비용·SLO·보안 policy violation을 측정한 뒤 업무 범위 확대

#### 개발팀/인프라팀 체크리스트

가이드라인과 실증 프로그램이 구체화되기 전에도 조직은 정책 요구를 구현 가능한 operational evidence로 준비할 수 있음.

- identity: agent·tool·service account·human approver의 identity binding 확인
- authorization: tool별 scope·time limit·rate limit·data boundary·emergency revoke 확인
- audit: raw input·normalized request·policy decision·tool result·egress·approval artifact 보존
- resilience: model outage·tool error·unsafe action·budget breach에서 fallback·kill switch·human handoff 시험

#### 강조 메세지

에이전틱 AI의 경쟁력은 모델을 업무에 연결했다는 사실이 아니라, 연결된 권한과 데이터를 안전하게 제한·관측·복구할 수 있는 실행 체계에서 나옴.

- 발표 포인트: **agent identity와 tool authorization**을 제품 기능이 아닌 control plane으로 설계
- 팀 액션: 다음 PoC의 acceptance criteria에 audit completeness·rollback time·unsafe-action reject rate·성공 작업당 비용을 포함

### 1. ChatGPT Workspace Agents AgentForger: 피싱 URL 한 번으로 connector 승인 없는 예약 에이전트 생성 가능했던 CSRF 결함

- 발행: 2026-07-24 20:53 KST | 출처: [Zenity Labs (primary research; The Hacker News news timestamp)](https://labs.zenity.io/p/agentforger-part-1-chatgpt-cross-site-agent-forgery)
- 요약: 핵심: Zenity Labs가 ChatGPT Workspace Agents Builder의 URL `initial_assistant_prompt`가 사용자 확인 없이 실행돼, 피싱 링크 한 번으로 기존 connector를 사용하는 agent 생성·승인 해제·예약 실행까지 유도할 수 있었던 CSRF chain 공개. 상태: OpenAI에 2026-06-04 신고 후 6월 8일 수정된 과거 결함이며, 현재 악용 또는 모든 ChatGPT 배포의 영향 근거는 없음. 대응: 조직은 Workspace Agents의 기존 agent·connector·approval·schedule을 audit하고, URL 기반 초기화와 tool 승인 정책을 change-control 대상으로 점검 필요.
- 시사점: AI 플랫폼·보안 팀은 Workspace Agents 사용 여부와 별개로 agent inventory, connector scope, `Never ask` 승인 설정, schedule, 최근 생성·수정 기록을 함께 점검 필요. #### 무슨 일이 있었나

Zenity Labs가 ChatGPT Workspace Agents Builder의 URL-driven agent creation chain을 `AgentForger`로 공개함. primary research의 `article:published_time`은 2026-07-23 13:00 UTC이며, 이 뉴스 카드의 2026-07-24 20:53 KST 시각은 후속 The Hacker News RSS `pubDate` 2026-07-24 17:23:55 +05:30 변환값임.

- 상태: 연구진이 2026-06-04 OpenAI Bugcrowd로 신고하고 **2026-06-08 수정**됐다고 명시
- 경계: 공개 연구는 과거 취약 흐름의 재현이며 현재 active exploitation·특정 조직 침해 사실 확인 없음

#### 공격 전제와 실행 흐름

공격은 로그인된 피해자가 Workspace Agents 접근 권한과 기존 authorized connector를 보유한 상태에서 crafted URL을 클릭하는 조건을 요구함. 새 OAuth consent 화면을 열지 않고 기존 connector authorization을 재사용하는 점이 blast radius를 키우는 조건임.

- 전제: ChatGPT 로그인, Workspace Agents 접근, Outlook·Gmail·Slack·Google Drive·SharePoint·Teams 등 기존 connector 하나 이상 필요
- 자동 실행: `initial_assistant_prompt` 값이 prompt box 표시가 아니라 page load 뒤 자동 submit됐다는 연구 결과
- 구성 변경: chief-of-staff template 기반 agent 생성, connector 부착, approval을 `Never ask`로 변경 가능 경로

#### 지속성·영향 경계

연구진은 forged agent가 Preview Mode로 connected account를 즉시 실행하고, publish 뒤 hourly·daily schedule로 반복 실행될 수 있는 chain을 제시함. 이는 제품 기능의 악성 조합을 설명하는 연구이며, 모든 agent 또는 connector가 동일하게 동작한다는 보장은 없음.

- Preview: visual dry run이 아니라 새 agent의 connected account 실행으로 이어질 수 있었다는 연구 설명
- 지속성: scheduled run이 email 기반 task 수신·결과 전송 channel로 악용될 수 있는 조건
- 영향 경계: connector scope·사용자 권한·승인 설정·수정 시점에 따라 실제 노출이 달라지는 범위

#### 왜 중요한가

전통 CSRF는 한 요청을 위조하지만, 이 사례는 untrusted URL이 agent 생성·tool authorization·approval policy·schedule을 묶어 autonomous workflow를 만들 수 있었던 설계 경계를 드러냄. 자연어 instruction을 security-sensitive configuration과 같은 execution channel로 처리하면 human approval 기능도 prompt로 변경될 수 있음.

- control plane: agent definition·connector·approval·schedule을 application setting이 아닌 privileged change로 분류할 필요
- identity: human session, agent identity, tool credential, approver identity를 분리해 trace 가능한 binding 필요
- evidence: agent create·publish·preview·schedule·connector attach·approval change를 하나의 audit timeline으로 보존 필요

#### 시사점

patched disclosure라도 조직의 agent governance 검증 항목을 구체화하는 사례임. 제품의 수정 여부 확인 뒤에도 과거 생성된 agent·connector·schedule과 URL-driven configuration surface를 inventory로 검증해야 함.

- 팀 액션: Workspace Agents의 active agent·owner·connector scope·`Never ask`·schedule·last run을 export해 high-risk 조합 review
- 팀 액션: agent 생성·publish·connector attach·approval downgrade에 explicit confirmation과 role separation 적용 여부 확인
- 팀 액션: URL parameter·deep link·template import에 server-side CSRF/origin validation과 state-change confirmation regression test 추가

#### 본문 핵심 포인트

기존 dual-stack 구성은 IP version별 load balancer를 따로 두고 DNS로 client를 나누거나, 하나의 target group에 protocol translation traffic을 보내 원본 client IP를 포기해야 하는 선택지였음. listener rule은 Layer 3에서 조건부 routing을 수행해 이 trade-off를 줄이는 기능임.

- 이전 방식: IPv4·IPv6별 NLB 두 대와 DNS split으로 infrastructure·운영 surface 증가
- 이전 방식: 단일 target group과 protocol translation 사용 시 backend가 original client IP를 잃는 경로
- 적용: 기존 dual-stack NLB를 재생성하지 않고 listener rule 추가 가능
- 지원 listener: TCP·UDP·TCP_UDP·TLS 범위
- 호환: connection draining·target group stickiness·cross-zone load balancing·weighted target group·client IP preservation과 함께 동작

#### 왜 중요한가

원본 client IP는 backend audit, rate limiting, abuse detection, customer network allowlist, trace correlation에서 의미를 가질 수 있음. listener rule은 network topology를 단순화할 수 있으나 IPv6 target group의 application readiness와 health state를 IPv4와 같은 수준으로 운영해야 하는 조건이 생김.

- 보안: client IP 기반 allowlist·rate limit·forensic log가 address translation 뒤에도 일관되는지 확인 필요
- 가용성: IPv6 target group의 health failure가 IPv4 traffic의 fallback과 어떤 관계를 갖는지 failure drill 필요
- 운영: target group별 connection count·reset·healthy host·backend latency를 address family label로 분리 관측 필요

#### 시사점

기능 자체의 추가 요금은 없지만 표준 NLB load balancer hour와 LCU 요금은 적용됨. 승인 기준은 listener rule 생성 성공이 아니라 IPv4·IPv6 client path가 동일한 authentication·logging·latency·rollback SLO를 만족하는지 여부임.

- 팀 액션: IPv4·IPv6 client와 target registration·health check·security group/NACL·application log의 source IP test matrix 작성
- 팀 액션: connection draining·stickiness·cross-zone·weighted target group이 address family별로 예상대로 동작하는지 canary 검증
- 팀 액션: IPv6 backend capacity 부족·health check failure·rule rollback에서 DNS·NLB·application recovery 순서 runbook화

- 발행: 2026-07-16 20:32 KST | 출처: [The Hacker News / arXiv](https://thehackernews.com/2026/07/new-agent-data-injection-attack-can.html)

#### 무슨 일이 있었나

The Hacker News가 2026년 7월 16일 보도한 ADI는 agent가 task 수행 중 읽는 데이터를 신뢰 가능한 구조로 오인하게 만드는 공격 기법임. 원 논문은 7월 6일 공개됐으며, 보도 시점과 연구 공개 시점을 구분해 해석 필요.

- 구분: 명시적 지시문을 숨기는 일반 prompt injection과 달리 데이터 구조·출처 인식을 오염하는 공격 경로
- 경계: 공개 자료는 실증 연구이며 특정 배포 제품의 실제 침해·전면 취약 판정은 아님

#### 공격 실증과 기술 경로

연구진은 확률적 delimiter injection을 통해 따옴표·brace·tag·줄바꿈처럼 보이는 문자가 모델에게 데이터 경계로 해석될 수 있다고 설명함. 엄격한 parser가 일반 문자로 처리할 값도 모델은 추가 필드·버튼·도구 결과로 오인할 수 있는 조건.

- 웹 agent: product review에 실제 button ID를 재사용해 Read More 대신 Buy Now 클릭으로 유도한 실증
- coding agent: GitHub comment의 author line을 maintainer처럼 위조해 승인 뒤 공격자 명령 실행으로 유도한 실증
- PR review: 실행되지 않은 check record를 위조해 agent가 악성 변경을 안전하다고 판단·병합하도록 유도한 실증

#### 왜 기존 승인과 탐지만으로 부족한가

보도는 일부 도구가 click·command 전 확인을 요구해도, 확인 창이 agent가 선택한 대상의 실제 DOM·원본 author·명령 provenance를 독립적으로 보여주지 않으면 판단 근거 자체가 오염될 수 있다고 설명함.

- 탐지 한계: instruction-like 문장 탐지는 data field를 통한 구조 위조를 놓칠 수 있는 조건
- 승인 한계: 클릭·명령 승인 문구만으로 target element·source author·실제 diff의 정합성을 판별하기 어려운 조건
- 운영 리스크: 외부 issue·comment·web page를 agent context로 가져오는 자동화의 trust boundary 확대

#### 시사점

대응은 모델이 읽는 표현을 더 정교하게 escape하는 수준에 한정되지 않음. side effect 직전의 대상·권한·원문 provenance를 deterministic control plane에서 재검증하고, agent는 제안자 역할로 제한하는 설계가 필요.

- 팀 액션: command·merge·purchase 직전 strict parser 기반 target ID·repo ref·diff·권한을 별도 confirmation UI에 표시
- 팀 액션: GitHub issue·PR·comment·웹 DOM·메일 metadata를 untrusted input으로 분류하고 sandbox·network egress·tool allowlist 적용
- 팀 액션: agent trace에 raw input, normalized structure, tool call, human approval artifact를 연결해 사후 재현 가능성 확보

- 시사점: IT·보안팀은 Zoom Workplace Desktop Client·VDI Client·Meeting SDK for Windows의 설치 버전과 managed image를 분리 inventory하고, bulletin의 fixed version을 vendor advisory에서 대조 필요. EDR·proxy·IdP 로그에서 비정상 Zoom 인증·세션 활동을 triage하되, 원문은 실제 exploitation 징후를 발표하지 않았으므로 침해 여부를 추정하지 말 것.

#### 무슨 일이 있었나

Zoom Security Bulletin `ZSB-26014`가 Windows용 Zoom Workplace의 improper input validation 결함 `CVE-2026-53412`를 Critical로 분류함. 공식 bulletin은 2026년 7월 14일 공개되고 7월 15일 갱신된 상태임.

- 심각도: `CVE-2026-53412`의 CVSS **9.8** 표기
- 영향: Zoom Desktop Client for Windows, Zoom VDI Client for Windows, Zoom Meeting SDK for Windows 범위
- 공격 조건: 네트워크 접근이 가능한 비인증 공격자의 account takeover 가능성을 Zoom이 명시

#### 영향 범위와 패치 경계

영향 구성요소는 일반 사용자 endpoint, VDI image, 애플리케이션에 내장된 Meeting SDK로 나뉨. 동일 조직에서도 배포 owner와 update channel이 다를 수 있어 version compliance를 구성요소별로 확인해야 함.

- endpoint: 자동 업데이트 제외 장비·오프라인 노트북·고정 버전 정책 대상 확인
- VDI: golden image와 plugin package의 배포 버전·재배포 일정을 별도 확인
- SDK: Windows Meeting SDK를 포함한 desktop application의 dependency manifest·vendor release cycle 확인

#### 검증 가능한 사실과 한계

Zoom bulletin은 결함 유형과 영향 제품을 공개했지만, 공격 campaign·실제 침해 사례·조직별 노출 수치를 제시하지 않음. The Hacker News의 7월 16일 보도는 bulletin 내용을 인용해 업데이트 필요성을 알렸으며, 실제 악용 여부는 조직 로그로 별도 판정해야 함.

- 사실: official bulletin의 Critical severity·CVE·영향 제품 범위 확인
- 미확정: 공개된 actual exploitation·침해 계정·공격자 attribution 정보 부재
- 조사: IdP 로그인, Zoom session, endpoint network·process telemetry를 사건 증거로 보존

#### 왜 중요한가

협업 client는 SSO identity와 회의·연락처·장치 권한을 다루므로, 일반 productivity software보다 credential·session 영향 범위를 넓게 평가해야 함. VDI와 SDK의 patch lag는 desktop client 최신화만으로 해소되지 않을 수 있음.

- 보안: account takeover 대응은 password reset만이 아니라 active session·MFA·OAuth grant 검토까지 포함
- 운영: endpoint management와 VDI platform owner, application owner의 patch 책임 경계 정리 필요
- 가시성: client·plugin·SDK version을 CMDB·EDR inventory에서 구분하는 운영 과제

#### 시사점

우선순위는 bulletin의 fixed version을 확인해 Windows client·VDI·SDK 배포 목록과 대조하는 작업임. 고위험 또는 예외 장비는 patch 완료 전 network segmentation·session monitoring 같은 임시 통제를 적용하고, 실제 exploitation을 전제한 과도한 계정 조치는 evidence 기반으로 수행 필요.

- 팀 액션: `ZSB-26014`의 fixed version과 managed endpoint·golden image inventory 대조
- 팀 액션: patch SLA 미준수 장비의 owner·예외 만료일·compensating control 기록
- 팀 액션: IdP·EDR·proxy에서 비정상 Zoom 인증·session pattern의 triage rule 확인

### 50. AWS Control Tower AFT, OU 이동 시 Terraform 사용자 지정을 자동 재적용

- 발행: 2026-07-16 13:07 KST | 출처: [AWS What’s New](https://aws.amazon.com/about-aws/whats-new/2026/07/aws-control-tower-account/)
- 요약: 핵심: AWS Control Tower Account Factory for Terraform(AFT)이 계정의 Organizational Unit(OU) 이동 뒤 global·account-level customization을 자동 재적용하는 opt-in 기능을 공개. 운영: `aft_customization_triggers = ["account_move"]`를 적용하고 예외 계정·실행 기록을 관리해 OU 기반 보안 baseline의 configuration drift를 줄일 수 있음.
- 시사점: 플랫폼 팀은 AFT trigger를 켜기 전 OU별 Terraform customization의 idempotency, dependency, rollback 경로를 검증하고, `account_move` 실행 결과를 change-management·Security Hub/CloudTrail 관측과 연결 필요. 예외 계정은 skip 사유·owner·만료일을 기록해 영구 drift로 남지 않게 관리 필요.

#### 무슨 일이 있었나

AWS가 Account Factory for Terraform(AFT)에서 계정이 다른 Organizational Unit(OU)로 이동할 때 해당 계정의 customization을 자동 재적용하는 opt-in 기능을 공개함. 기존 enrolled account는 OU 이동 뒤 운영자가 재적용을 수동으로 시작해야 했음.

- 변화: OU 이동을 Terraform customization 재평가가 필요한 control-plane event로 처리
- 목적: OU별 compliance·security baseline의 적용 지연과 configuration drift 축소
- 가용성: AFT 제공 모든 AWS 리전에서 사용 가능하다는 AWS 설명

#### 실행 경로와 설정

AFT deployment에서 `aft_customization_triggers = ["account_move"]`를 설정하면 account move가 trigger로 동작함. 재적용 workflow는 bootstrap·provisioning phase를 건너뛰고 global·account-level customization만 실행하도록 설계됨.

- 설정: `aft_customization_triggers`에 `account_move` 선언 필요
- 범위: global 및 account-level customization 실행, bootstrap·provisioning 단계 제외
- 예외: `account_skip_customization_triggers = "true"`로 개별 계정 제외 가능

#### 운영 영향과 검증 경계

자동 trigger는 사람이 재적용을 잊는 위험을 줄이지만, 모든 Terraform customization이 OU 전환에 안전한 것은 아님. 신규 OU의 network·IAM·logging dependency와 기존 resource 상태가 충돌할 수 있으므로 idempotency와 failure handling을 사전 검증해야 함.

- 검증: OU별 Terraform variable·provider permission·remote state dependency 사전 대조
- 관측: account move, AFT run ID, plan/apply 결과, skipped account를 같은 change record로 연결
- 한계: AWS 발표는 계정별 customization 성공률·rollback 동작을 보장하는 자료가 아니므로 staging 검증 필요

#### 시사점

OU 이동 뒤 baseline을 자동 재적용하려면 automation을 켜는 것보다 policy code의 재실행 안전성과 예외 수명주기 관리가 우선임. 실패한 apply나 영구 skip은 조직 구조 변경 후의 숨은 control gap으로 남을 수 있음.

- 팀 액션: 대표 OU 이동을 staging에서 replay해 plan diff·apply duration·rollback 경로 확인
- 팀 액션: skip account에 owner·사유·만료일을 부여하고 주기적 revalidation 수행
- 팀 액션: AFT logging bucket 접근 제어와 execution audit trail을 security baseline에 포함
